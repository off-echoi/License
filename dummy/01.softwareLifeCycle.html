1. 요구사항 확인

001. 소프트웨어 생명 주기

  • 소프트웨어 개발 방법론의 바탕
  • 소프트웨어 개발 정의, 운용, 유지보수의 과정을 각 단계별로 나눈 것
  • 개발자는 문제유형이나 개발방법에따라 특정모형을 선택할 수 있다.

  1. 폭포수모형

    • 각 단계를 확실히 매듭짓고 검토, 승인 => 다음단계
    • 전통적인, 고전적 생명주기 모형
    • 선형 순차적 모형
    • 메뉴얼 작성 필수
    • 두개이상의 과정이 병행수행 되지 않음
    • 타당성검토 계획 요구분석 설계 구현 시험 유지보수

  2. 프로토타입모형(Prototype Model, 원형 모형)

    • 견본을 만들어 결과물 예측
    • 사용자와 시스템사이의 인터페이스에 중점을 두어 개발
    • 구현단계에서의 골격코드
    • 폭포수의 단점보완
    • 요구수집 빠른설계 프로토타입구축 고객평가 프로토타입 조정 구현 다시 처음으로 순환 > 요구수정 > 빠른설계...

  3. 나선형모형(Spiral Model, 점진적 모형)

    • 보헴, 폭포수 + 프로토
    • 나선을 돌듯이 소프트웨어 개발을 거치며 점진적으로 완벽한 최종 소프트웨어 개발
    • 위험 관리 및 최소화
    • 정밀, 유지보수가 필요없음
    • 계획 및 정의 위험분석 공학적 개발 고객평가 순환...

  4. 애자일 모형(Agile Model)

    • 민첩한, 기민한 고객의 요구사항 변화에 유연하게 대응
    • 짧은 일정주기반복
    • 고객과의 소통에 중점
    • 스프린트, 이터레이션이라는 짧은 개발주기 반복 > 주기마다 고객의 평가와 요구 수용
    • 소규모 프로젝트, 숙달된 개발자, 급변하는 요구사항
    • 스크럼, XP, 칸반, FDD 등
    • 핵심가치 : 프로세스·도구 X 상호작용 O , 문서 X 실행되는 SW o, 계약 X 협업 O , 계획 X 변화 O

  5. 폭포수와 애자일 비교

    구분 폭포수 애자일
    새로운 요구사항 반영 어려움 지속적 반영
    고객 의사소통 적음 많음
    테스트 마지막 일정주기
    개발중심 계획, 문서 고객

002. 스크럼 기법

  • 팀원 스스로 스크럼 팀 구성, 개발작업에 관한 모든것을 스스로해결
  • 제품책임자, 스크럼마스터, 개발팀

  • 제품책임자(PO)

    제품이해도 높, 의사결정,요구사항 작성 주체, 개발의뢰자 or 사용자
    요구사항 백로그 작성 , 우선순위 지정
    팀원 : 백로그 작성가능하지만 순위 X
    테스트하며 주기적으로 요구사항 갱신

    스크럼 마스터(SM)

    객관적으로 조언/ 팀원통제아님!
    일일 스크럼 주관 장애요소 공론화 => 처리

    개발팀(DT)

    제품책임자, 스크럼마스터제외 모든 팀원 대상
    최대 7~8명

  • 스크럼개발 프로세스

    제품 백로그(product backlog)

    • 요구사항을 우선순위에 따라 나열
    • 지속적으로 업데이트
    • 요구사항기반 전체 일정 계획 릴리즈

    스프린트 계획 회의(sprint planning meeting)

    • 스프린트 주기 단기일정 수립
    • 작업단위 분할, 개별 스프린트 백로그 작성

    스프린트(Sprint)

    • 2~4주 개발 진행
    • 태스크대상 작업시간 추정 > 담당자 배정
    • 개발자가 원하는 테스크 담당
    • Todo, Inp Pogress, Done 상태를 갖는다.

    일일 스크럼회의(Daily Scrumm Meeting)

    • 매일 15분 서서진행, 소멸차트에 표시
    • 스크럼마스터는 장애요소 해결 서포트

    스프린트 검토회의(Sprint Review)

    • 사용자 테스팅
    • 스프린트 한주당 한시간내
    • 제품책임자는 피드백정리 > 백로그 업데이트

003. XP(eXtreme Programming)

  • 고객참여와 개발과정 반복
  • 짧고 반복적 개발주기, 단순설계, 고객참여 > 빠른개발 목적
  • 릴리즈기간 짧게 반복 > 요구사항의 가시성 높힘
  • 소규모
  • 의사소통, 단순성, 용기, 존중, 피드백

  • XP 개발 프로세스

    사용자스토리(user story)

    • 고객의 요구사항 시나리오
    • 기능단위로 구성

    릴리스계획 수립(Release Planning)

    • 부분 혹은 전체 개발완료 시점 일정 수립

    스파이크(spike)

    • 위험감소를 위한 별도의 프로그램
    • 오직 처리할 문제만 작성(다른조건 무시)

    이터레이션(iterlation), 주기

    • 릴리즈 세분화 단위
    • 1~3주, 새로운 스토리작성 > 이터레이션 포함

    승인검사(Accept Test), 인수 테스트

    • 이터레이션 내의 릴리즈 제품 완료, 테스트
    • 오류는 다음 이터레이션

    소규모 릴리즈(Small Release)

    • 릴리즈기간동안 이터레이션이 완료(테스트, 릴리즈)되면 최종결과물 전달
    • 릴리즈가 완성품이 아닌결루 계속 진행

004. UML(Unified Modeling Language)

  • 개발자와 고객사이의 의사소통을 위한 표준화된 객체지향 모델링 언어
  • 시스템 구조 : 6개의 구조 다이어그램 / 시스템 동작 : 7개의 행위 다이어그램
  • 사물, 관계, 다이어그램

  1. 사물(Things)

    • 다이어그램안에서 관계가 형성될 수 있는 대상
    • 사물 내용
      구조 사물
      Structural Things
      • 개념적, 물리적 요소
      • 클래스, 노드, 컴포넌트, 유스케이스
      행동사물
      Behavioral Things      		 시간, 공간에따른 요소들의 행위
      상호작용, 상태머신
      그룹사물
      Grouping Things      		 그룹으로 묶은 요소
      패키지
      주해사물
      Annotation Things      		 부가적인 설명이나 제약

  2. 관계(Relationship)

    • 사물과 사물사이의 집합
